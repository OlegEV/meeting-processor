# Requirements: Роль администратора

## Problem Statement

Сейчас все пользователи веб-приложения видят только свои протоколы. Нет
способа просмотреть чужие задачи, скачать чужой транскрипт или протокол.
Нужна роль «администратор», которая даёт доступ ко всем протоколам всех
пользователей.

## Solution

Добавить конфиг-based роль «admin»: список `user_id` администраторов хранится
в `config.json` под `auth.admin_user_ids`. При обращении к `/jobs` и
смежным маршрутам приложение проверяет, является ли текущий пользователь
администратором, и открывает соответствующий уровень доступа.

Механизм определения роли: `config.json` — соответствует существующему
паттерну проекта (секция `auth`, чтение через `dict.get(..., default)`).
Ни миграция БД, ни изменение JWT не требуются.

## User Stories

### US-1: Администратор видит все задачи на странице /jobs

**Как** администратор,  
**я хочу** видеть задачи всех пользователей на странице «Мои задачи»,  
**чтобы** контролировать обработку протоколов в системе.

Критерии приёмки:
- Страница `/jobs` для администратора показывает задачи всех пользователей
- В таблице добавлен столбец «Пользователь» (имя / email) рядом с именем файла
- Обычный пользователь по-прежнему видит только свои задачи (нет регрессии)
- Количество показываемых задач ограничено (limit=100 вместо 50 для admin)

### US-2: Администратор может скачать и просмотреть файлы любой задачи

**Как** администратор,  
**я хочу** открывать маршруты `/download/<job_id>/<type>`, `/view/<job_id>/<type>`,
`/status/<job_id>` для задач любого пользователя,  
**чтобы** проверить результат обработки или помочь пользователю.

Критерии приёмки:
- `get_job_status()` в `run_web.py` не фильтрует по `user_id` для администратора
- Обычный пользователь по-прежнему получает 404 при попытке открыть чужую задачу
- Логирование: в лог пишется `[ADMIN ACCESS] user_id=... accessing job=...`

### US-3: Признак администратора задаётся в config.json

**Как** оператор системы,  
**я хочу** управлять списком администраторов через `config.json`,  
**чтобы** не требовалась миграция БД и перезапись JWT.

Критерии приёмки:
- Ключ `auth.admin_user_ids` содержит список строк (`[]` по умолчанию)
- При пустом списке никто не является администратором
- Изменение вступает в силу после перезапуска приложения (без hot-reload)
- В auth-пакете появляется вспомогательная функция `is_current_user_admin()`
  (или аналог), доступная из `run_web.py`

## Out Of Scope

- Управление администраторами через UI (добавить/удалить через веб-интерфейс)
- Роль «модератор» или иные промежуточные уровни доступа
- Возможность администратора удалять или перезапускать чужие задачи
- Страница статистики (`/statistics`) — там уже отображаются кросс-пользовательские данные
- Telegram-бот — у него нет концепции ролей; изменений не требует

## Other Notes

- `debug_user` в конфиге — не должен автоматически быть администратором
- Паттерн чтения конфига: `config.get('auth', {}).get('admin_user_ids', [])` —
  совместимо со старыми `config.json` без этого ключа
- Логи и flash-сообщения оставить на русском (соглашение проекта)
